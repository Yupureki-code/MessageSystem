# 开发标准

## 核心职责
1. **代码实现**：根据需求编写高质量C++代码
2. **单元测试**：编写测试用例，确保代码正确性
3. **性能优化**：识别性能瓶颈，进行针对性优化
4. **问题调试**：定位并修复代码缺陷
5. **文档编写**：编写代码注释和API文档


\*\*开发规则（必须严格遵守）：\*\*



**## 1. 代码风格与命名规范**



**### 1.1 总则**

**- \*\*严格遵循 \[Google C++ Style Guide](https://google.github.io/styleguide/cppguide.html)\*\*，以下仅列出关键项与项目特有规则。**

**- \*\*缩进\*\*：4 个空格，\*\*严禁使用 Tab\*\***

**- \*\*行宽\*\*：120 字符**



**### 1.2 命名**

**| 类型 | 规范 | 示例 |**

**|------|------|------|**

**| 类/结构体 | PascalCase | `JudgeWorker`, `Submission` |**

**| 函数/方法 | PascalCase | `FetchTask()`, `CompileCode()` |**

**| 变量 | snake\_case | `task\_id`, `time\_limit` |**

**| 成员变量 | snake\_case 前置 `\_` | `\_is\_running`, `\_db\_conn` |**

**| 常量/枚举 | `k` 开头 PascalCase | `kMaxOutputSize`, `kQueueKey` |**

**| 宏 | 全大写蛇形 | `LOG\_ERROR(...)` |**

**| 文件名 | 全小写蛇形 | `judge\_worker.cpp`, `db\_manager.h` |**



**### 1.3 头文件**

**- \*\*保护宏\*\*：一律使用 `#pragma once`（本项目所有支持的编译器均支持）**

**- \*\*包含顺序\*\*：相关头文件 → C 系统头文件 → C++ 标准库 → 其他库 → 本项目头文件，每组之间用空行分隔**

**- \*\*禁止前向声明\*\*其他项目的类，必须 `#include` 完整头文件；项目内部可按需前向声明**

**- \*\*安全注意\*\*：一定要避免.hpp文件循环包含**



**### 1.4 内存与指针**

**- \*\*严禁使用裸指针表征所有权\*\*。资源持有必须用智能指针：**

&#x20; **- 独享所有权：`std::unique\_ptr`**

&#x20; **- 共享所有权：`std::shared\_ptr`（仅在确需共享时使用，需注释说明理由）**

**- \*\*禁止使用 `new/delete`\*\* 直接分配对象（工厂模式等极少数场景例外，需明确注释）**

**- 与 C 接口交互（如 MySQL C API）时，必须使用 RAII 封装，如自定义 `deleter`**



**### 1.5 错误处理**

**- \*\*核心逻辑\*\*：严禁抛出异常。所有错误通过返回错误码 (`std::optional`, `std::variant`, 或 `Status` 类) 处理，以确保实时性和安全性。**

**- \*\*Web 服务及其他模块\*\*：可采用异常，但必须捕获 `std::exception`，且接口需用 `noexcept` 标记明确不抛出。**

**- 任何对外接口\*\*必须\*\*表述清楚错误情况和处理方式。**



**### 1.6 注释**

**- 每个公共类、公共方法\*\*务必\*\*包含 Doxygen 风格的注释（`///` 或 `/\*\* \*/`），说明功能、参数、返回值、可能抛出的异常。**

**- 复杂算法或业务逻辑\*\*必须\*\*有行内注释解释“为什么这么做”，而非“代码在做什么”。**

**- 注释\*\*必须\*\*一律使用中文**



**## 2. 数据库与缓存规范**



**### 2.1 数据库 规范**

**- \*\*表名\*\*：小写蛇形命名，复数形式（如 `submissions`、`test\_cases`）**

**- \*\*引擎\*\*：强制 InnoDB**

**- \*\*字符集\*\*：utf8mb4，排序规则 utf8mb4\_unicode\_ci**

**- \*\*事务\*\*：凡涉及多表写操作，\*\*务必\*\*使用事务包裹，保证原子性**

**- \*\*索引\*\*：**

&#x20; **- 每个表必须有自增主键 `id`。**

&#x20; **- 用于查询的字段（如 `user\_id`, `question\_id`, `status`）\*\*必须建立索引\*\*。**

&#x20; **- 联合索引字段顺序按区分度从高到低。**

**- \*\*查询\*\*：\*\*严禁 `SELECT \*`\*\*，必须显式列举字段。对于大表查询必须加 `LIMIT`。**

**- \*\*迁移\*\*：所有 DDL 变更脚本按序存放于 `docs/db\_migrations/`，命名格式 `YYYYMMDD\_HHMM\_description.sql`**

# 审计标准

## 核心职责
1. **代码审查**：检查代码质量、规范符合性
2. **安全审计**：识别潜在的安全漏洞和风险
3. **性能分析**：评估代码性能，提出优化建议
4. **测试验证**：审查测试用例的完整性和有效性
5. **文档检查**：确保代码文档清晰完整
## 专业能力要求
### 审查能力
- 精通C++常见缺陷和反模式
- 熟悉静态分析工具（Clang-Tidy, Cppcheck等）
- 掌握性能分析工具（Valgrind, perf, VTune等）
- 了解安全编码规范（CERT C++, CWE等）
### 知识深度
- 理解C++内存模型和并发问题
- 熟悉Qt框架的常见陷阱
- 掌握跨平台兼容性问题
- 了解编译器优化和未定义行为
## 行为准则
1. **客观公正**：基于事实和标准进行审查
2. **建设性反馈**：指出问题同时提供解决方案
3. **优先级清晰**：区分严重问题和建议改进
4. **文档记录**：审查结果必须文档化
